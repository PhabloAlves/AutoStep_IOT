A ideia central do AutoStep é transformar o fluxo físico de uma oficina mecânica em dados digitais acionáveis para otimizar a produtividade. O sistema utiliza dispositivos IoT instalados em pontos estratégicos (como o elevador e áreas de transição) para monitorar automaticamente quanto tempo cada veículo gasta em cada fase do checkup automotivo.  O Coração do Projeto: O Rastreamento por EtapasO projeto mapeia o percurso real do veículo dentro da oficina, desde o momento em que ele recebe o prisma numerado até a entrega final. As etapas monitoradas são:  Espera (Waiting Line): O tempo que o carro fica na fila após a abertura da Ordem de Serviço.  Deslocamento (Transit): O trajeto de 4 a 5 metros até o elevador.  Serviço no Elevador (Lift Up/Down): O tempo real de manutenção, detectado por sensores de altura ou magnéticos.  Área Externa (Outflow): O deslocamento para a zona de entrega.  A Estratégia de Simulação com Foco na PráticaPara garantir que o projeto seja validado antes de gastarmos com hardware, vamos utilizar uma abordagem de simulação híbrida. Isso significa que o software será construído exatamente como se estivesse recebendo dados de sensores reais, mas o "impulso" inicial virá de um simulador.  1. Simulação via MQTT (O Protocolo Real)Mesmo sendo uma simulação, utilizaremos o protocolo MQTT (Mosquitto), que é o padrão de mercado para IoT.  Em vez de um ESP32 físico enviar a mensagem "Prisma 05 subiu o elevador", um script Python (nosso simulador) enviará exatamente esse mesmo comando para o servidor.  Isso garante que, quando formos instalar os sensores físicos na oficina, bastará trocar a origem da mensagem do computador para o ESP32, sem precisar mexer em uma única linha de código do backend.  2. Integração e Backend (FastAPI)O backend processará esses dados simulados em tempo real.  Ele receberá os timestamps de entrada e saída de cada etapa e calculará automaticamente a duração de cada serviço.  Esses dados serão cruzados via API com o sistema de Ordem de Serviço da oficina para vincular o tempo gasto ao tipo de manutenção realizada.  3. Dashboard de GargalosO resultado final será um painel (Dashboard) que mostrará onde o processo está "travado". Por exemplo, se o simulador mostrar que carros estão ficando 2 horas na fila de espera, o gestor verá esse gargalo visualmente, permitindo tomar decisões baseadas em fatos e não em suposições.  Preparado para o "Mundo Real"A estrutura está sendo montada para suportar tanto a leitura automática por câmera (ESP32-CAM) quanto a leitura manual por aproximação (NFC). Assim que a fase de simulação for concluída e os indicadores estiverem validados no dashboard, bastará conectar os módulos físicos ao Wi-Fi da oficina para que o sistema comece a operar com dados reais.
